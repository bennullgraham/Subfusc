Stack

This site is cached by varnish. Behind varnish, nginx either serves static files directly or hands off application requests to gevent. gevent serves a lightweight Python application written in the Flask web framework.

Flask-Assets is employed to bundle and filter CSS and Javascript source files. Javascript is run through rJsmin, and CSS through pyScss, a Python implementation of the compiler for the SCSS variant of Sass. All Javascript source is loaded asynchronously with a variant of $script.js, with the exception of one or two small inline snippets (one of which is, obviously, the asynchronous loader itself).

Performance

Ignoring transport-layer delays, the stack employed allows thousands of requests per second with response times in the tens of milliseconds. These figures were obtained with: 

$ sudo httperf --hog --server=bgraham.com.au --uri=/ --num-conns=5000 --rate=2500 --num-calls=3

While the server may respond incredibly quickly to requests, the majority of delay from a user perspective is caused by the network round-trip time - i.e., the time it takes for a request to physically traverse the Internet. The order and loading of resources has been optimised to lower these delays to the absolute minimum. The chart alongside shows the network timeline for a request made in Melbourne, Australia to the server hosting this site - an EC2 instance in Tokyo, Japan. The blue vertical line represents the DOMContentLoaded event, which, roughly speaking, is when the page begins to display. The bare minimum of resources are requested before allowing this event to occur, meaning the delay is approximately double the network round-trip time. This is essentially the best attainable delay for a web page. Other resources are then requested, and the page is 'finished' at the red vertical line.

Design

The design employed here is understated and emphasises readability - I hope. There are clear vertical lines for the eye to follow (note that all text abutts a single left rule), while horizontal spacing respects a certain rhythm. Content is also readable on mobile or small-display devices without serving a different design - try resizing your browser window. The content will adjust gracefully all the way down to "mobile portrait" format, thanks to Skeleton.

Deployment

Development changes are pushed to a bare repository on the production server:

$ git push live

In that repository, a post-receive hook pushes to the directory the site is served from:

# hooks/post-receive: 
#!/bin/sh
GIT_WORK_TREE=/var/www/subfusc/ git checkout -f

This deployment method is very lightweight. Site customisation derives from the application loading config defaults but allowing an environment variable which specifies overrides in a local config file:

app = Flask(__name__)
app.config.from_object('default-config')
if os.environ.get('SUBFUSC_SETTINGS'):
    app.config.from_envvar('SUBFUSC_SETTINGS')

default-config.py contains configs appropriate for production. SUBFUSC_SETTINGS is actually used to tailor the site for a development environment, e.g.:

# site-config.py
DEBUG = True
 
$ export SUBFUSC_SETTINGS="$(pwd)/site-config.py"
$ python Subfusc.py
